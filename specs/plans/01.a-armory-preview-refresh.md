# Armory Preview Refresh Implementation Plan

## Overview
Elevate the armory preview so the player’s chicken hero appears with the currently equipped cosmetic, and hovering/equipping upgrades instantly shows the matching visual/audio feedback. We’ll build low-poly assets that mirror the concept reference, wire a lightweight Three.js preview mounted in the HUD, and play short ability/VFX loops per item without compromising performance.

## Current State Analysis
- `packages/client/src/game/hud.ts:375` renders the preview area purely with DOM/CSS placeholders; no 3D canvas or animation exists.
- The main renderer (`packages/client/src/game/bootstrap.ts`) already defines procedural chicken meshes and gameplay VFX, but they aren’t reusable in the armory.
- Armory inventory data (`packages/server/src/index.ts:165`) provides text/stats only; no visual metadata ties upgrades to effects.
- Audio helpers (`packages/client/src/game/audio.ts`) expose UI-friendly stingers, but the preview doesn’t trigger them.

## Desired End State
- Preview stage runs a performant Three.js scene featuring the player’s chicken with cosmetic attachments, matching upgrade hover/equip state.
- Ability previews run a brief loop (0.5–1.5 s) showing signature VFX, then settle into idle until the next interaction.
- Audio cues fire on hover/equip, respecting the existing user-gesture gate.
- Assets remain within agreed triangle/material budgets and pause gracefully when the HUD hides or window loses focus.

## What We’re NOT Doing
- No enemy showcase or squad multi-character display.
- No overhaul of gameplay renderer or networking payloads.
- No new asset loading pipeline (keep assets code-authored or lightweight inline textures).
- No cosmetic purchase logic changes on the server.

## Implementation Approach
Author cosmetic/ability attachments once, expose them via a new preview asset module, and drive a dedicated `ArmoryPreviewRenderer` that reuses Three.js primitives with disciplined update loops (30 fps cap, paused when hidden). HUD events will orchestrate the preview state machine, and documentation will spell out how to extend the system for future cosmetics.

## Phase 2: Preview Renderer Integration
### Overview
Embed a dedicated Three.js renderer within the HUD preview stage, sharing lighting/material settings and throttled updates.

### Changes Required
- **File**: `packages/client/src/game/armoryPreviewRenderer.ts`
  - Implement `ArmoryPreviewRenderer` class that:
    - Accepts the HUD stage element, mounts a `<canvas>`, and spins up `WebGLRenderer` (`alpha: true`, DPR-capped).
    - Creates its own `Scene`, `PerspectiveCamera`, ambient + directional lights matching `(160,340,180)`.
    - Provides `loadRig`, `applyCosmetic`, `playAbilityPreview`, `setIdle` methods.
    - Caps render loop to 30 fps with `requestAnimationFrame` + delta gating; pauses when `disconnect()` called.
- **File**: `packages/client/src/game/hud.ts:375`
  - Replace static glyph with renderer mount hook (`previewStage.appendChild(renderer.canvas)` and lifecycle calls).
  - Listen for `window.resize` to relay dimensions to renderer.
- **File**: `packages/client/src/game/bootstrap.ts`
  - Pass existing `audio` reference into HUD options so preview can trigger SFX.
