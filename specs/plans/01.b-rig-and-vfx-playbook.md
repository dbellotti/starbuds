# Rig & VFX Playbook Foundations Implementation Plan

## Overview
Establish shared rigging, cosmetic attachment, and VFX authoring standards so future playable characters and enemies can plug into the armory preview (and broader client renderer) with minimal bespoke code. The goal is to codify skeleton schemas, reusable effect presets, and supporting tools/documentation.

## Current State Analysis
- Existing chicken rig in `packages/client/src/game/bootstrap.ts` is procedural and undocumented; no shared schema describes bone names, attachment sockets, or animation expectations.
- Upgrade/VFX mappings are hard-coded per feature; there is no catalog of reusable shader presets, palettes, or effect parameters.
- Artists/developers lack a checklist covering triangle budgets, userData expectations, and naming conventions when preparing new low-poly assets.
- Tooling to preview individual rigs/effects outside the game loop does not exist, making iteration slow as more characters arrive.

## Desired End State
- A documented rig schema (types + checklist) covering skeleton joints, attachment sockets, tintable submeshes, and animation expectations for both heroes and enemies.
- A data-driven VFX library defining reusable presets (e.g., psionic pulse, ember trail) with tunable color/intensity, referenced by upgrade or character definitions.
- Authoring SOP outlining asset budgets, required metadata, and export steps for designers/artists.
- Optional CLI/sandbox script to load and snapshot rig/effect definitions for quick iteration outside the armory.

## What We’re NOT Doing
- No overhaul of existing in-game animation systems; only establishing standards/interfaces.
- No commitment to migrate current chicken rig immediately—retrofits can happen incrementally.
- No new gameplay logic or networking changes.
- No armory UI work beyond providing references for future integrations.

## Implementation Approach
Create a small shared package for rig/effect definitions, back it with structured TypeScript types, and produce living documentation. Seed the library with current chicken + planned enemy archetypes, and provide lightweight tooling to preview or lint new definitions.

## Phase 1: Rig Schema & Definitions
### Overview
Codify skeletons, sockets, and base metadata for playable characters/enemies.

### Changes Required
- **File**: `packages/shared/src/rigs.ts`
  - Define `RigDefinition`, `SocketDefinition`, `AnimationChannel` types and export canonical joints (e.g., `wing.L`, `crest`, `tail`) for heroes and enemies.
- **File**: `packages/client/src/game/rigRegistry.ts`
  - Provide lookup functions (`getRigDefinition(id)`) and helper utilities to validate rigs at runtime.
- **Docs**: `docs/assets.md`
  - Document triangle budgets, required sockets, naming conventions, and tint metadata expectations.

```ts
// packages/shared/src/rigs.ts
export interface RigDefinition {
  id: string;
  joints: Record<string, JointSpec>;
  sockets: SocketDefinition[];
  tintTargets: string[];
  animationChannels: AnimationChannel[];
}
```

### Success Criteria
#### Automated Verification
- [ ] `npm run typecheck --workspaces`
- [ ] `npm run lint`

#### Manual Verification
- [ ] Rig definitions cover chicken hero + at least one enemy archetype
- [ ] Documentation reviewed with art/engineering stakeholders
- [ ] Runtime validation logs missing sockets when loading legacy rigs (opt-in)

## Phase 2: VFX Preset Library
### Overview
Create reusable effect presets and tie them to upgrade/character definitions.

### Changes Required
- **File**: `packages/shared/src/vfxPresets.ts`
  - Declare preset metadata (`colorPalette`, `geometry`, `shaderId`, `duration`) keyed by logical names (e.g., `psionicPulse`, `emberTrail`).
- **File**: `packages/client/src/game/vfxRegistry.ts`
  - Map presets to factory functions that instantiate Three.js meshes/shaders; expose `createEffect(presetId, context)`.
- **File**: `packages/shared/src/index.ts`
  - Add optional `vfxPresetIds` property to `ArmoryItem` or new `VisualDescriptor` type for future use.
- **Docs**: Update `docs/art-style.md` with palette guidance referencing preset IDs.

```ts
// packages/shared/src/vfxPresets.ts
export const VFX_PRESETS: Record<VfxPresetId, VfxPreset> = {
  psionicPulse: {
    baseColor: 0x60a5fa,
    glowColor: 0x8ecaff,
    shader: 'radialPulse',
    duration: 1.1
  },
  emberTrail: { /* ... */ }
};
```

### Success Criteria
#### Automated Verification
- [ ] Typecheck/lint clean

#### Manual Verification
- [ ] Preview sandbox can instantiate each preset without errors
- [ ] Palettes/emissive levels align with art direction feedback
- [ ] Future armory items can reference presets instead of hard-coded values

## Phase 3: Tooling & Preview Sandbox
### Overview
Deliver a lightweight way to inspect rigs/effects outside the game loop and validate metadata.

### Changes Required
- **Tool**: `tools/preview-rig.ts` (Node script or simple Vite page)
  - Load a rig/effect definition and render it in isolation, exporting screenshot/GIF option.
- **Script**: `package.json` root
  - Add `npm run tools:preview-rig -- <rigId>` to launch the sandbox.
- **Optional**: `tools/lint-rigs.ts`
  - Validate asset metadata (triangle counts, required sockets) from exported JSON manifests if available.

```ts
// tools/preview-rig.ts
import { getRigDefinition } from '../packages/client/src/game/rigRegistry';
// Spin up minimal Three.js scene, display rig with sample animation, allow screenshot export.
```

### Success Criteria
#### Automated Verification
- [ ] Typecheck/lint cover tooling (use tsconfig references if needed)

#### Manual Verification
- [ ] Running `npm run tools:preview-rig -- chicken-hero` opens sandbox and shows rig + default animation
- [ ] Tool supports switching VFX presets for quick visual checks
- [ ] Lint script flags missing sockets/tint metadata on intentionally malformed sample
